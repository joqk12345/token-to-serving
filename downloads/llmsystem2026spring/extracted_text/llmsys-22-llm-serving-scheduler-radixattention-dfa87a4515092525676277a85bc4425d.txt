• Optimizing for Throughput and Latency

• Utilize heterogeneous devices (CPU/GPU)

          Inside Scheduler of SGLang
• get_next_batch_to_run()
  o if finishing prefill: moving the completed prefill to decode batch
  o if new prefill request: get_new_batch_prefill
     ▪ takes a new request, depending on the available spot in batch, and
       waiting queue of requests (calculating the priority of requests), then
       addPrefill
  o if no new prefill: continue the decode update_running_batch()
     ▪ check_decode_mem: is there enough GPU memory for running
        – yes ➔ prepare_for_decode, reduce new_token_ratio
        – no => batch.retract_decode, increase new_token_ratio, put the req back to
          prefill_waiting_list
  o process_batch_result: check if any req is complete. free the kv
    cache (only reference, not actually free mem)                                     16

            KV Cache Management
• Standard NN inference only stores the current layer’s
  output, but do not store all intermediate results for each
  layer

• This is ok for LSTM/RNN, but it is increasingly slower for
  Transformer inference – generating each next token would
  require all previous keys and values for each Transformer
  layer.

• Store all keys, values for each layer in GPU memory ➔ KV

Managing KV Cache with RadixAttention
• KV mem pointers are stored in a radix tree (i.e. prefix tree)
  o each edge is a string
  o each node contains memory pointers of KVs
  o the whole path’s string represent the prefix

• Retrieve KV cache for a prompt string
  o lookup a prompt string’s longest prefix from the radix tree: very
    fast by matching the edge
  o returns: number of tokens in prefix matched, the matched node
